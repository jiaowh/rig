00 — the problem

The backwards problem

Physics runs one way. Manufacturing needs it to run the other way.

Suppose you grow a thin crystal film in a machine. You choose a recipe: heat the substrate to 1,340 K, run the growth for long enough to deposit 800 nanometres. The machine runs. Hours later, metrology tells you what you got: the film is 2.1% non-uniform across the wafer, it bowed 4 µm on cooldown, the centre hit 1,318 K.

That direction — settings → results — is the forward direction. It's what physics simulators do, and it's what a neural network naturally learns.

But nobody in a fab actually wants that. A process engineer starts from the other end. They have a specification: non-uniformity must be under 1.5%, bow under 5 µm, thickness 800 nm ± 20. Their question is the backwards one:

The question this program answers

Given the outcome I need, what recipe should I run? And — because being wrong costs a real wafer, a real chamber-hour, and real money — how confident should I be, and when should you tell me it can't be done at all?

Running the backwards question is called inverse design, and it is not merely "the forward model in reverse". It is harder in four specific ways, and every design decision in this program is a response to one of them.

Why you can't just invert the function

1. Many recipes give the same answer (non-injectivity). A cooler substrate for longer and a hotter one for less time can land on the identical thickness. So "the" inverse doesn't exist — there is a set of answers, called the pre-image. A program returning one recipe is hiding the truth from you. A useful program returns several genuinely different ones, so you can pick on cost, or robustness, or whatever it doesn't know about.

2. Some answers are lies the model tells itself. This is the big one. If you ask an optimiser to find settings the model predicts will hit your target, it will happily walk into a region where no data was ever collected, where the model is confidently, catastrophically wrong, and hand you that. It looks perfect on paper. It destroys a wafer. This failure mode has a name — reward hacking the surrogate — and defending against it is most of the work.

3. The process is noisy. Run the same recipe twice, get two different wafers. So a recipe that lands exactly on the spec limit is a coin-flip, not an answer. You need room to spare.

4. The knobs drift. You dial 1,340 K; the tool delivers 1,341.8 K. A recipe that only works at exactly the dialled value isn't a recipe, it's a fluke.

Hold onto those four. The rest of this document is, in order, the machinery that answers each one.

01 — vocabulary

The five words you need

Everything below is built from these. They're worth two minutes.

WordWhat it meansConcretely, in this program
Recipe The settings you control. Written x. Substrate temperature 1150–1500 K; film thickness 0.2–5 µm. These are the only things the inverse is allowed to search over.
Outcome What you measure afterwards. Written y. Non-uniformity %, centre temperature, bow in µm, thickness grown. Several numbers at once, not one.
Spec box The acceptable region of outcomes. Written Z*. A box, never a point: "bow between 0 and 5 µm". A point target is rejected outright — no recipe can hold a safety margin around a single exact value.
Forward model A learned stand-in for the machine that predicts y from x with error bars. Returns four things every time — see below. Never a bare number.
Inverse solver Takes a spec box, returns recipes — or refuses. Returns a ranked, deliberately diverse set, each with a confidence and a feasibility flag; or an explicit INFEASIBLE verdict with a diagnosis.

One structural point that matters more than it looks. Every prediction in this system is this object, and nothing is allowed to return anything less:

PredictiveDistribution( mean — the best guess aleatoric_sigma — irreducible noise: the process itself is random epistemic_sigma — the model's own ignorance: fixable with data conformal_set — an interval with a *guaranteed* hit rate )

Keeping those two sigmas apart is not pedantry — it's the hinge the entire program turns on. Section 03 explains why.

02 — stage one

Learning the process from very little data

In a fab, every training example costs a wafer. You will not get a million of them. You might get forty.

This constraint kills the obvious answer. A deep network trained on forty points will overfit, and — worse for our purposes — it won't know that it's ignorant away from those points. So the default engine here is a Gaussian process (GP), which is the right tool for exactly this regime: it fits smooth functions from a handful of points, and it reports honestly-growing uncertainty as you move away from data.

The idea behind a Gaussian process

Instead of fitting one curve, a GP represents every curve that could plausibly explain your data, weighted by plausibility. Predicting at a new recipe means asking: across all those plausible curves, what's the average value here, and how much do they disagree?

Where you have data, the curves are pinned together and they agree — low uncertainty. Far from data, they fan apart — high uncertainty. That fanning is the thing we'll later exploit.

What holds it together is the kernel, a function saying how similar two recipes are. This program uses Matérn-5/2:

k(x, x′) = σf² · (1 + √5·r/ℓ + 5r²/(3ℓ²)) · exp(−√5·r/ℓ) where r = |x − x′|, the distance between two recipes
ℓ
lengthscale — how far you must move before the outcome changes appreciably. Small ℓ = twitchy process.
σf²
signal variance — how much the outcome varies overall.
ARD
Each knob gets its own ℓ ("automatic relevance determination"). A knob with a huge lengthscale barely matters — the model learns which knobs are irrelevant, and tells you.

Why Matérn-5/2 and not the more famous squared-exponential? The latter assumes the process is infinitely smooth — cartoonishly so. Matérn-5/2 assumes twice-differentiable: bendy but able to have kinks. Real processes have kinks. This is a deliberate, conservative choice.

The actual prediction equations

Given training recipes X and measured outcomes y, the prediction at a new recipe x* is closed-form — no gradient descent, no training loop:

K = k(X, X) + σn²·I similarity of training points to each other, + noise k* = k(X, x*) similarity of the new point to each training point μ(x*) = k*ᵀ · K⁻¹ · y → the mean σ²(x*) = k(x*,x*) − k*ᵀ · K⁻¹ · k* → the epistemic variance

Read the mean as a weighted average of outcomes you've already measured, weighted by how similar their recipes are to the one you're asking about. Read the variance as: start at total prior ignorance, subtract whatever the training data explains. Near data, the subtraction is large and uncertainty collapses. Far away, it subtracts nothing and you're left with full ignorance. That's the entire mechanism, and it's why a GP can't fool itself the way a network can.

The hyperparameters (ℓ, σf², σn²) are fitted by maximising the marginal likelihood — with analytic gradients that match a finite-difference check to 3×10⁻⁹.

03 — the hinge

Two kinds of not-knowing

Conflate these and the program is worthless. Separate them and everything else follows.

Suppose the model says 2.0% non-uniformity, give or take 0.4. There are two utterly different reasons for that 0.4.

Aleatoric — the dice

The process is genuinely random. Same recipe, different wafer, different answer: thermal fluctuation, flux jitter, metrology noise. More data will not reduce this. It's a property of the world. From alea, Latin for dice.

Epistemic — the ignorance

The model hasn't seen this part of the space. More data removes this entirely. It's a property of your knowledge, not the world. From episteme, Greek for knowledge.

Why the distinction earns its keep — it answers two different questions:

  • High epistemic → "go run an experiment here, you'd learn something." This drives which experiment to run next (§08).
  • High aleatoric → "an experiment here teaches you nothing new; this region is just noisy. Leave margin or avoid it."

A model reporting one lumped error bar cannot tell you which situation you're in — so it can neither choose experiments intelligently nor be appropriately paranoid. That's why the canonical prediction object above keeps them in separate fields, and why that's enforced by a test.

Play with it below. Watch the two bands behave completely differently as you add data.

04 — calibration

Making the error bars honest

A model's "90% interval" is a claim. Conformal prediction turns it into a guarantee.

The GP's error bars rest on assumptions — Gaussian noise, correct kernel, well-fitted hyperparameters. When those are wrong, the bars are wrong, usually too narrow, and a "90%" interval quietly covers 70% of the time. In a fab, that discrepancy is scrapped product.

Conformal prediction fixes this with a trick that is almost embarrassingly simple and comes with a real mathematical guarantee that does not care whether your model is any good.

How it works, in four steps

  1. Hold out some runs the model never trained on — the calibration set.
  2. On each, measure how wrong the model was, scaled by how wrong it said it might be: score = |y − μ(x)| / σ(x).
  3. Sort those scores. Take the one at the ⌈(1−α)(n+1)⌉-th position — for 90% coverage, roughly the 90th percentile. Call it q̂.
  4. Every future interval is μ(x) ± q̂·σ(x).

If the model was overconfident, the observed scores are large, q̂ comes out big, and every future band gets stretched until it actually covers. The guarantee holds under one assumption — that future runs look statistically like the calibration runs (exchangeability) — and needs nothing else. Not Gaussianity. Not a correct kernel. Not even a competent model.

The detail everyone gets wrong

That ⌈(1−α)(n+1)⌉ is not a rounding nicety. When it exceeds n — too few calibration points to justify the claim — the honest answer is an infinitely wide interval, and this program returns exactly that rather than a comfortable-looking finite band it cannot back. With 8 points you cannot make a 90% claim; the code says so out loud. (Verified against exact rational arithmetic across α ∈ {0.01…0.5} × n ∈ [1, 2000]: zero off-by-ones.)

The scorecard for this is PICP — Prediction Interval Coverage Probability — the fraction of held-out runs that actually landed inside their band. Aim 90%, measure 90%. On the real sputtering dataset this program scores 0.929 against a 0.90 target. (§10 has an important caveat about how that coverage is bought.)

When the process drifts

Split conformal assumes tomorrow's runs look like yesterday's — exactly the assumption a drifting tool breaks. So the program carries an online tier that re-earns its coverage run by run, scoring each new interval before observing the outcome and then adjusting. Two calibrators do this: ACI (adaptive conformal inference), which nudges the miscoverage level α after every hit or miss — with one known vice: pushed hard enough it can "cover" by emitting an infinitely wide interval; and conformal-PID, the designated endpoint, which tracks the score threshold itself with a proportional-plus-integral controller and is finite by construction — it cannot buy coverage with an unbounded band. Both run with fixed library defaults, identical for every dataset, and a rolling coverage window doubles as the drift detector, so a drift episode is flagged even while the average coverage recovers. §10 shows both at work on real data.

05 — stage two

The pessimistic inverse

This is the heart of the program. Everything before it was preparation.

Here's the naïve approach, and why it's dangerous. You want the outcome in [L, U]. So you search for a recipe where the model's predicted mean lands in [L, U]. Done.

And it will hand you a catastrophe, for three compounding reasons:

  • The mean is just a guess. If it's on the edge of the box, real noise pushes you out half the time.
  • The model might be badly wrong here — and the optimiser is actively drawn to those regions, because that's where the model makes its most flattering predictions. It hunts for its own blind spots.
  • The tool won't deliver exactly what you dialled anyway.

So this program inverts the logic. Rather than asking "where does the model predict success?", it asks:

The actual objective (§8.1)

Where would we still land in spec even if everything that could plausibly be against us, is? Assume the model is wrong in the worst direction it admits it could be. Assume the tool drifts the worst way within tolerance. Then demand room to spare on top. A recipe only survives if it's still in spec after all of that.

Building the margin, one hostile assumption at a time

Start with the model's prediction at recipe x for output j: mean μⱼ, epistemic σ_epi,ⱼ, aleatoric σ_ale,ⱼ. Now displace the mean by everything that could hurt:

sⱼ(x) = z_epi · σ_epi,ⱼ(x) ① the model might be wrong — shove μ toward the wall + Σᵢ |Jⱼᵢ(x)| · Δᵢ ② the knobs drift — worst case over the tolerance box

Term ② deserves a second. Jⱼᵢ = ∂μⱼ/∂xᵢ is the Jacobian — how hard output j reacts to knob i. Multiply by that knob's tolerance Δᵢ and you get the worst shift it can cause. Absolute value, because we assume it drifts the bad way. Sum over knobs, because they can all conspire at once. Mathematically this is the exact first-order worst case over a box of drift — and it means a recipe sitting on a knife-edge slope gets penalised automatically. Sensitivity is risk, and this term prices it.

Now measure the remaining room to each wall, in units of process noise:

u_hiⱼ = (Uⱼ − μⱼ − sⱼ) / σ_ale,ⱼ standardized room below the ceiling u_loⱼ = (μⱼ − Lⱼ − sⱼ) / σ_ale,ⱼ standardized room above the floor FEASIBLE ⟺ minⱼ min(u_hiⱼ, u_loⱼ) ≥ κ
κ
How many noise-widths of spare room you demand. Default 2.0. This is ③ — the answer to the dice.
z_epi
How paranoid about model error. Default 2.0.
Δ
Knob tolerance, default 2% of each range.
minⱼ
Over every constrained output — one failure is failure. No averaging away a violation.

Dividing by σ_ale is what makes κ meaningful. It says: express the remaining room in units of how much this process randomly wobbles. Somewhere noisy, you need a physically wider gap to clear the same κ. Betting where the process is unpredictable gets penalised automatically, without a special rule.

A subtlety worth flagging

Epistemic uncertainty enters exactly once, in s. An earlier version of the specification also subtracted a separate κ·σ_epi term — double-counting the same fear and making the solver so timid it declared everything impossible. The code carries a comment specifically forbidding the double-count. Conservatism has to be deliberate, not accidentally stacked.

The anti-reward-hacking gate

One hole remains, and it's the nastiest. What if the model's epistemic uncertainty is spuriously small in a far-away region it never saw — a blind spot it doesn't know is a blind spot? Then s is small, the margin looks luxurious, and the pessimism was pessimism about nothing.

So there's a second, independent test: support score. It asks "does this recipe even look like the runs we've done?" — computed as a negative Mahalanobis distance from the training cloud (0 at the centre, increasingly negative as you leave). Any candidate below the floor — by default the 5th percentile of the training data's own scores — is rejected outright, no matter how good its margin looks.

And this gate fails closed: the solver's constructor refuses to be built without either a support floor or the training data to derive one. You cannot forget to turn on the safety, and you cannot switch it off. That is a deliberate design choice, and it is genuinely load-bearing: it is the gate that keeps a mis-scaled search space from returning confident nonsense.

The calibrated backstop

The κ-margin above reads the model's raw σ values — so it is only as honest as the model's own opinion of its error, and pessimism inherits any miscalibration underneath it. The program closes that loop with §04's machinery: whenever the model carries calibrated conformal bands, every candidate must also pass the containment check C(x) ⊆ Z* — the calibrated interval, not just the raw-σ margin, must fit inside the spec box — and this runs by default, with the same no-false-abstention sweep as the rest of the solver. Every returned recipe carries its calibration provenance: conformal-checked or revalidated when a calibrated model vouched for it, model-feasible when only the raw margins did — which is labelled precisely because it is not a calibrated guarantee.

Returning a set, not an answer

Look at Figure 3 again: the certified region is often several disconnected islands. That's non-injectivity — problem ①. Handing back one recipe would silently discard genuine alternatives that might be cheaper or more robust in ways the model can't see.

So the solver returns a diverse set, chosen by farthest-point selection: anchor on the highest-confidence recipe, then repeatedly add whichever survivor is farthest from everything picked so far. That deliberately reaches across to other islands instead of returning four neighbours from one. And if the pre-image really is a single point, it returns one candidate and stops — it never pads the list with near-duplicates to look thorough.

Each returned candidate carries a confidence, computed as the probability of landing in spec under the worst-case mean:

confidence = Πⱼ ( Φ(u_hiⱼ) + Φ(u_loⱼ) − 1 ) Φ = normal CDF

The product treats outputs as independent — an approximation the code documents rather than hides (a proper joint-residual-covariance version is scheduled work). Being explicit about which corners are cut is a recurring habit in this codebase, and a good sign.

06 — refusal

Saying "impossible" properly

The most valuable output of an inverse solver is often a well-explained no.

A system that always returns something is a system that lies when the honest answer is "your target is unreachable". So INFEASIBLE here is a first-class result type, not an exception and never a clipped best-effort point:

Infeasible( nearest_achievable the closest we CAN get distance_to_feasible how far short, in σ reason *why* — and this is the useful part )

The why matters because "impossible" has four completely different causes demanding four completely different responses. Telling them apart needs a trick: the pessimistic search deliberately avoids high-uncertainty regions, so it never even visits the point that would be feasible if the model were confident. The solver therefore runs a second, epistemic-free probe — the same search with model-paranoia switched off — purely to diagnose. Comparing the two searches gives the verdict:

DiagnosisWhat it meansWhat you should do
Off-manifoldEvery candidate failed the support gate — the target lives where you've never operated.Expand the explored region first. You're asking about terra incognita.
Epistemic-limitedFeasible if the model were confident. Ignorance is the only thing blocking you.Collect runs. The target is real; you just haven't earned it yet.
Partly epistemicMean lands in the box, but the box is narrower than the noise band.Data will help some, but you'll also need to relax the spec.
Hard conflictThe mean can't reach the box at all, even optimistically.No amount of data fixes this. Change the target, or change the process.

That probe does one more job: if it stumbles on an on-support, genuinely-feasible recipe that the main search missed, it promotes it. The rule is absolute — never return INFEASIBLE when a feasible recipe exists.

A subtle trap this rule closes

The diverse picks are chosen by spread, not by merit — so when a re-validation or containment gate rejects them, their failure says nothing about the dozens of other certified candidates still in the pool. Abstaining at that point would be a false "impossible", the exact failure this section forbids. So the solver never does: before concluding INFEASIBLE it sweeps the entire remaining certified pool, highest-confidence first, and only abstains when nothing survives.

07 — legality

Never proposing an illegal recipe

A neat trick that makes an entire category of bug impossible.

The heater runs 1150–1500 K. An optimiser exploring freely will absolutely try 1600 K. The usual fix is to clip — but clipping creates flat spots that confuse the optimiser, and it silently piles candidates onto the boundary.

Instead: change coordinates so illegal values don't exist. The optimiser works in an unconstrained variable u ranging over all real numbers, and every u is mapped into the legal box:

x = lo + (hi − lo) · sigmoid(u) sigmoid squashes ℝ into (0,1) u = −∞ → x = lo u = 0 → x = midpoint u = +∞ → x = hi

The optimiser can propose any number, however wild, and the recipe that comes out is always legal. Not "checked and rejected" — incapable of being illegal. The constraint stops being a rule to enforce and becomes a property of the coordinate system.

Compositions get the same treatment. Alloy fractions must be non-negative and sum to exactly 1 — a fiddly constraint. Under x = softmax([u, 0]) both hold automatically for any u, and the inverse uᵢ = log(xᵢ/x_K) makes it a true bijection.

08 — stage three

Choosing the next experiment

You have budget for forty runs. Which forty?

Recall the moment in Figure 3 where adding six runs rescued an infeasible target. That's the whole thesis of the active-learning loop: the model tells you what it doesn't know, and you go buy exactly that knowledge. Automating it needs a rule for scoring a candidate experiment before you run it.

Two ways to value an experiment

BALD asks: how much would running here reduce my ignorance about the process in general? It's the epistemic-to-aleatoric ratio — high where the model is unsure but the process isn't inherently noisy. It refuses to be lured into noisy regions where you'd learn nothing:

BALD(x) = ½ · log(1 + σ_epi²(x) / σ_ale²(x)) in nats

EPIG asks a sharper question: how much would running here reduce my ignorance about the recipes I actually care about? It uses the GP's ability to say how correlated two locations are. Learning at x teaches you about your target x* exactly insofar as they're correlated — so EPIG will send you somewhere convenient that happens to be informative about somewhere expensive.

Blend them, and divide by cost:

α(x) = [ λ · EPIG(x) + (1 − λ) · BALD(x) ] / cost(x)^β

Both terms are in nats — genuine information units — which is what makes adding them legitimate rather than mixing incompatible scales. Cost divides rather than subtracts, so the score reads as information per unit money.

Then two schedules anneal over the campaign. λ: 0.2 → 0.9 — start by exploring broadly, finish by aiming hard at the goal. β: 1 → 0 — start cost-conscious, finish willing to pay for the run that closes it out. Early cheap experiments buy the map; late expensive ones buy the destination.

Each batch spends one run exploiting (the inverse's best guess — which also tests whether you're done) and the rest exploring, chosen to be diverse so you don't buy the same information four times. The model is refit from scratch each round, and the loop stops on target hit, budget exhaustion, or stall.

A consistency rule this design pins

EPIG mixes two quantities from the model — its prediction and its joint covariance. If those two ever describe different probability distributions, the mixture breaks an underlying inequality and the information gain silently collapses toward zero: the score is never wrong, just absent, which is worse. So every forward model in the program must satisfy the identity EPIG(x; {x}) ≡ BALD(x) — learning about a point equals its own uncertainty — which holds for any self-consistent model and is pinned by test on every tier.

09 — stage four

The instant answer

Optimisation takes seconds. Sometimes you need microseconds.

Everything so far solves one target by running a fresh optimisation. Good answers, but you pay each time. The amortized generator pays once, up front, then answers instantly — the difference between solving an equation and looking one up.

The tool is a normalizing flow. The idea: start with pure Gaussian noise and push it through a learned, invertible warp until it takes the shape of your target distribution. Because every step is invertible, you can both sample from it and compute exact densities. Trained conditionally on the spec box, it learns "show me recipes for this target" directly.

Two design choices do a lot of work:

  • It's trained in the unconstrained u-space of §07 — so every sample it ever draws is a legal recipe by construction. A flow can't be taught to respect a box; it doesn't need to be.
  • Several flows are ensembled. Where they disagree, that's epistemic uncertainty showing up in the proposals themselves.

The two halves then combine — the flow proposes cheaply, the §05 solver refines and certifies. The measured result: the same quality as a cold 32-start search for 28% of the compute.

The gate that guards it

A generator that produces confident nonsense is worse than none. So it can't ship without passing Simulation-Based Calibration. The logic is elegant: if the posterior is truly calibrated, then for a randomly drawn true recipe, its rank among the posterior's samples should be uniform. Test the flow against itself many times and check that uniformity. A calibrated flow passes; an undertrained one is caught.

The binding rule behind this gate

The gate must certify the law that actually ships. The calibration gate and the sampler draw from the same even mixture over ensemble members, by construction — because any daylight between what the gate tests and what the product ships means the certificate describes an object the user never receives. Any change to how sampling works must be mirrored, in the same change, in what the gate validates. That symmetry is a written, binding rule of the codebase, pinned by test.

10 — the evidence

What it demonstrably does

Everything below is current, reproduces from a seed, and is scored against ground truth — never against the model's own opinion of itself.

Three rules govern every number here. Ground truth: an inverse is scored by running the returned recipe on the true process, never by asking the surrogate that proposed it. Determinism: every recorded result regenerates byte-for-byte from its seed, and each was run twice to prove it. Provenance: headline claims are computed only on real-tool data; simulation is stamped as simulation and barred from masquerading (§12).

On real tool data — the Empa HiPIMS campaigns

The full pipeline — generic ingest, GP forward model, conformal calibration, coverage gate, pessimistic inverse — runs end-to-end on 3,150 measured runs from a real magnetron sputter tool: six campaigns, aluminium and titanium targets across power tiers (§12 has the dataset's card). The coverage gate asks one question per campaign: does the claimed 90% interval actually cover 90% of held-out runs — on a genuine run-order temporal split, with a shuffled split as the exchangeable control?

The same data settles the material question. A model trained on one campaign is provably blind to a change of target material — material is not a knob, so knob-space support and epistemic screening cannot see the shift. Conditioning a pooled model on material fixes what is fixable: the shift becomes representable (the predicted mean moves 0.8–1.7 Å/s where the blind model moved nothing), a query with unspecified material triggers dominating epistemic uncertainty every time, and pooling costs no coverage anywhere. What it does not deliver is cross-material transfer: trained on one material and shown 10–20 runs of the other, prediction error stays 2–14× the full-data ceiling with miscalibrated intervals. That claim is therefore forbidden — measured, not assumed away. Per-material models remain the honest configuration, with the material-aware pooled screen on top.

And the inverse behaves on real data exactly as §§05–06 promise. Asked for an over-tight band: refused, with a quantified diagnosis. Asked for a target beyond the data (−10.8σ): refused — "more data will not move it in." Asked for a creditable band: three recipes, each verified against the nearest actually-measured run, all landing inside the target band.

In silico — where the comparisons live

  • Against warm-started Bayesian optimization on the simulated machine (metrology noise on, matched budgets, bit-identical warm starts, scored on the simulator's own output): RIG reaches spec roughly 2× cheaper, wins at every tolerance tested, and delivers calibrated feasibility where the pessimism binds — 0% false-accepts against the baseline's 7.8%. The recorded sweep runs at a deliberately permissive margin policy and says so in its result file; a re-run at the binding policy is the current work item.
  • Dimensional reach: with the restart budget scaling by dimension, the solver returns ground-truth-verified recipes from 2 to 20 knobs — and at 20 knobs with too little data it does the more impressive thing: refuses, names "collect runs" as the cause, and is vindicated when doubling the data flips it to a genuine hit while 4× the search budget alone does not. The honest weakness at that scale: the raw-σ margin can run out of buffer when the surrogate's own error estimate is too rosy — which is exactly why the calibrated containment backstop of §05 sits on the default path, and why any answer certified without a calibrated model is labelled model-feasible.
  • The amortized tier (§09) trains, samples, and passes its calibration gate — but its acceptance evidence is still toy-scale; an acceptance run on the simulated machine under a stricter pass rule is the current work item. Until it lands, treat the instant-answer tier as machinery, not as a result.

The caveats that travel with the numbers

  • One real tool, one lab, optimization-clustered sampling. No multi-tool replication, no replicate recipes, no prospective confirmations yet.
  • The 0.929 coverage on the little sputtering grid (§04) is bought with inflated fitted noise — the model absorbs its misfit into "noise" until bands cover — and that dataset is a dense 2-knob lookup grid: a plumbing proof, not inverse evidence. Its inverse also only fires at tolerances wide enough to be nearly free.
  • A production release additionally requires independent confirmation runs on the real tool. The confirmation-batch gate and its campaign orchestration exist and are tested — but no real campaign has run them. Until one does, this is a research prototype that can propose and abstain; it is not a deployed recipe service, and this page does not claim otherwise.
11 — the comparison

How this compares to the published alternative

A Berkeley/Lam Research team published a different answer to the same problem. Here is where their approach genuinely beats ours, and where the physics of a real fab is on our side — argued from their own numbers, not ours.

The paper is “Few-Shot Test-Time Optimization Without Retraining for Semiconductor Recipe Generation and Beyond” (Gu et al., arXiv:2505.16060, May 2025), and its method is called Model Feedback Learning (MFL). The idea is elegant: train a tiny reverse neural network that maps a desired outcome straight back to a recipe, first against a fast emulator (Loop A), then fine-tune it against the machine itself (Loop B). Five machine iterations later it lands a recipe in spec — versus ~20 for a Bayesian-optimization-plus-human workflow and ~84 for a senior engineer. Those are real strengths, and two of them we simply concede below.

Same goal, opposite bets

Both systems answer “what inputs produce this outcome?”. They disagree on what an answer is.

The margin problem — in their own Table 1

MFL regresses to a point, and a point target has no concept of how close to the cliff edge it is. Their own headline result shows why that matters. The etch-depth spec floor is 2250 nm (below it, the wafer is out of spec). Their stated point target is 2260 — just 10 nm above that floor. Their method landed at 2255.55:

spec floor 2250
landed 2255.55
their target 2260
← out of spec remaining safety margin: 5.55 nm  |  their own convergence error: 4.45 nm in spec →
The method’s demonstrated error is 80% of its entire safety margin

MFL missed its own point target by 4.45 nm — and only 5.55 nm of margin remained to the hard floor. One more unit of the error it already demonstrably has, and the recipe is out of spec. Nothing in an MSE-to-a-point loss can see this coming, because “distance to the target” and “distance to the cliff” are different numbers, and MFL only measures the first. RIG’s §8 decision rule is the second number: FEASIBLE ⟺ minj marginj ≥ κ·σ. A recipe this close to the floor, with any honest noise estimate, does not get certified here — it gets refused, with the reason named.

The gradient problem — what “5 iterations” costs on a real machine

MFL’s machine-tuning loop (Loop B) forms its gradient by, in the paper’s own words, “replacing the emulator E with the true machine M” in their Eq. 4 — an expression that contains ∂M/∂x, the machine’s Jacobian. In their experiments M is a neural network, so the Jacobian is one free autograd call. A plasma etcher does not have autograd. On real hardware the only way to get ∂M/∂x is finite differences: with their own 11 input knobs, that is at least 12 wafer runs per gradient step, so “5 iterations” becomes ≈60 machine runs — and the paper’s headline comparison (5 vs. 20 vs. 84, baselines quoted from a different system’s study) quietly inverts. RIG is built to never need the machine’s gradient: every derivative it uses comes from its own surrogate, and the machine is only ever asked for values.

What MFL is genuinely better at

Concessions first — this is not a takedown

Amortization. One trained 7 kB reverse model answers new targets in a forward pass. RIG’s certified answer is an optimization run per query — minutes, not milliseconds. (RIG’s amortized generator — stage 09 above — is the counterpart, and unlike MFL it ships with a calibration gate that can refuse a miscalibrated generator; but its acceptance run so far is on a toy problem, as section 10 already told you.) Simplicity. Two small networks and two training loops; RIG carries a GP, conformal layers, a constrained solver, and gates. The two-loop idea itself. Pre-train on a cheap emulator, fine-tune on the expensive machine — that is a good, reusable design, and their ablation shows it earns its keep.

The head-to-head — pre-registered, steelmanned, and scored

We ran the comparison on RIG’s simulated MBE machine: 20 pre-registered targets (10 clearly feasible, 5 feasible-but-hard near the reachable boundary, 5 provably infeasible — labels verified by an independent 20,000-point search), identical seed data for both methods, everything scored against the machine’s noise-free ground truth. The predictions were frozen before any comparison code existed, including two where RIG was predicted to lose. The MFL arm was adversarially “steelmanned” first: an independent reviewer verified the implementation faithful to the paper (its training gradient matches the analytic form to 8×10⁻¹⁷) and confirmed no cheap tuning change makes it stronger — so a RIG win can’t be blamed on a weak opponent.

The frozen predictions scored 3-for-5. The certified-miss and edge-hugging predictions held (0.00 vs 0.50; +10.5σ vs −1.45σ). The predicted loss on false abstention also held: RIG refused all five feasible-but-hard targets — the full, honestly-paid price of pessimism. But the prediction that MFL would win on machine-run count was refuted in the opposite direction: RIG used 60 runs to MFL’s 4,060, because RIG’s surrogate answers inverse queries without touching the machine at all — the GP is the amortization. The structure of the result is a perfect split: both methods nail all 10 easy targets; at the boundary and beyond, RIG refuses while MFL confidently presents 10 recipes and misses all 10.

The referee’s footnote that applies to both

Neither method has touched a real machine

Their §5, verbatim: “the experiments are conducted in simulation.” Their “machine” is an MLP trained on Gaussian-sampled synthetic data — so their headline result is a neural net inverting another neural net. Every number in the table above is in-silico too — it compares the two methods on RIG’s simulator, not their plasma-etch task, and that caveat stays loud. On the argument that actually matters — a certified result on real tool data — RIG now has one (the coverage gates and real-data-verified recipes of section 10, on the dataset of section 12); MFL, by its own paper, has none. The head-to-head itself, though, remains a home-turf simulation comparison, and this page says so rather than pretending otherwise.

12 — the data

The data it runs on

Three sources, three jobs: a simulator to develop against, one small public file that proves the plumbing, and one real dataset — the program's chosen venue — that carries every headline claim.

1 · The in-silico machine — development and comparisons

An external MBE physics simulator (mbe_sim, a sibling repository) wrapped in InSilicoMachine, which injects the unpleasant parts of a real tool — drift, first-wafer offsets, chamber seasoning, heteroscedastic metrology noise. Nothing is stored: data regenerates from seeds on demand, and every run is stamped provenance = physics_sim. The data contract bars that stamp from every headline claim, by design, so simulation can never masquerade as evidence. The method comparisons of §11 and the dimensional stress tests of §10 live here.

2 · A plumbing proof — the sputtering grid

209 measured zirconium magnetron-sputtering runs (source power × chamber pressure → deposition rates) from a public self-driving-lab study, jarlsanna/gps-for-magnetron-sputtering. It proved the real-data code path end-to-end, and can never be more than that: the repository states no license, so the file stays on-prem and uncited; and it is a dense 15×15 grid over two knobs — invertible by nearest-neighbour, so it cannot distinguish a learned inverse from a table lookup.

3 · The venue — Empa bipolar HiPIMS (real_tool)

The dataset behind every real-data result in §10, chosen by the process owner as the program's M0 venue.
